  Autointegrate, Architecture and Decision Record (draft)

  1. The constraints that drive the whole design

  Four requirements in the brief shape every decision. Stating them up front
  because the architecture is mostly a direct consequence:

  - No hardcoding. Endpoints, tenants, status values, field names, counts, none
  of it may live in code, prompts, tools, or seed memory. The only runtime input
  is the pasted spec. Consequence: the code knows exactly two things, how to 
  make an HTTP call and how to manage markdown files. Everything else is learned
  data on disk.
  - Fresh sessions must work. Steps 2 and 4 happen in new chat sessions with no
  memory of the teaching conversation. Consequence: durable knowledge cannot
  ride on the SDK's session-resume; it must live entirely on the filesystem and
  be re-loaded at the start of every process. We deliberately do not use
  resume/continue to carry knowledge across the teach‚Üíuse boundary.
  - Scoped learning. Customer-specific facts must not leak across tenants;
  integration-wide facts must carry across both. Consequence: the memory layout
  has two scope tiers, and self-correction must classify which tier a finding
  belongs to.
  - Triggered self-correction (your call). Realtime, deterministic trigger, no
  cron. Consequence: we use the SDK's PostToolUse / PostToolUseFailure hooks on
  the HTTP tool to fire correction, instead of relying on the model to notice on
  its own.

  context-rot worry: we never stuff all learned knowledge into context; the
  agent reads the slice it needs.

  Scoping rule (the heart of the multi-tenant requirement): when writing a
  correction, the agent classifies it.
  - Contract-level facts (real field names, the actual status enum, date format,
  pagination/limit behavior, endpoint quirks) ‚Üí integration.md, carries to all
  customers.
  - Distribution/config facts (how this tenant uses a status, a tenant-specific
  custom field, "acme's data looks like X") ‚Üí that customer's profile.md.
  - Default when unsure: scope to the customer, narrow blast radius. Promote to
  integration-wide only when it's clearly a contract fact or it's been observed
  on more than one customer. This is the safe direction: it prevents wrongly
  applying one customer's quirk to the other, which is the explicit failure the
  brief warns about.

  4. The three flows mapped to SDK features

  Teach (first contact, multi-agent probe). Operator pastes the spec. The ingest
  scaffold:
  1. Parses the spec into a draft config (trust the spec as a starting point).
  2. Spawns a small fan-out of probing subagents (SDK agents), each owning a
  slice: enumerate the list endpoint and test limit/pagination behavior; verify
  field schema reality vs spec; discover the real status enum; enumerate
  tenants; probe write endpoints minimally. Reads are free; writes are done
  cautiously and logged (see cuts).
  3. A synthesis step merges findings into integration.md + per-customer
  profile.md, recording every spec-vs-reality delta as the first corrections. So
  some correction happens proactively at ingestion, not only reactively at use
  time, both paths write the same files.

  Use (fresh session). Operator asks a question or gives an action for a
  customer. Agent reads INDEX.md, identifies the API + customer, reads the
  relevant integration.md + profile.md, then answers/acts via http_request. No
  teaching-session memory needed; the filesystem supplies everything.

  Correct (triggered). The PostToolUse / PostToolUseFailure hook on http_request
  fires on:
  - Hard triggers (deterministic): non-2xx status, network error, or response
  that structurally contradicts the learned config (unknown status value, a
  documented field missing, schema drift). Hook injects a tight, scoped
  instruction: "that call to <path> returned <X>; the config claims <Y>;
  investigate the real behavior and update memory/apis/<api>/‚Ä¶, tagging the
  scope."
  - Soft triggers (semantic, prompt-level not hook-level): operator pushback
  ("that's way too low") arrives as a normal turn; the system prompt makes
  operator skepticism authoritative evidence that current understanding is
  wrong, prompting re-derivation. Honest boundary: we do not claim to
  deterministically detect every implausible-but-valid answer; the hook catches
  structural drift, the operator/agent judgment catches semantic drift.

  The correction is written to disk in the same session, so a later fresh
  session reads the corrected file. That is how "it stays fixed" holds. Using a
  hook rather than model discretion is what makes this deterministic and 
  rot-resistant: it fires regardless of conversation length and doesn't depend
  on the model remembering to self-check, which were your two stated concerns
  with the fully-autonomous option.

  5. Key decisions and tradeoffs

  - Filesystem memory over a DB/vector store. Inspectable (the evaluator reads
  it), diffable, editable by the agent with the same Read/Edit/Grep tools, zero
  infra. Tradeoff: no query engine; we rely on grep/glob and a small index,
  which is fine at this scale.
  - Ephemeral SDK sessions, filesystem as the only durable state. Maps cleanly
  to the fresh-session requirement. Tradeoff: we give up free conversational
  continuity across runs, which we don't want here anyway.
  - Hook-triggered correction over model-discretion or cron. Deterministic +
  realtime + rot-resistant. Tradeoff: structural triggers are detectable,
  semantic ones need the operator or model judgment; we're explicit about that
  line.
  - Dedicated generic HTTP tool over Bash/curl. Gives the hook structured
  response access. Tradeoff: a bit more code than just enabling Bash.
  - Multi-agent probing on ingestion. Parallel coverage, each prober has a clean
  focused context. Tradeoff: subagents don't share history (must pass slices +
  merge via files) and cost more tokens; we keep the fan-out modest and can
  scale it.
  - bypassPermissions mode for the autonomous loop. Needed so it reads/writes
  memory and calls the API without prompting a human each turn. Tradeoff: full
  local system access; only safe on a dev machine/sandbox, and we can scope
  destructive Bash via disallowedTools and gate critical memory files via a

‚Źļ Memory updated. Here's how multi-transport folds into the design.

  Transport is a pluggable layer, not an assumption

  The single biggest "don't wire it to the sandbox" risk was assuming HTTP/REST.
  Fixing that with a transport-adapter seam:

  agent expresses intent ("get acme's applications where status=hired")
          ‚Ēā
          ‚Ėľ
   transport call tools  (the agent picks based on the LEARNED transport)
     http_request(method, path, query?, body?, headers?)
     graphql_request(query, variables?)
     grpc_request(service, method, message)
          ‚Ēā  each adapter resolves connection (base URL/endpoint, auth) from
  active config
          ‚Ėľ
   normalized response envelope  { ok, status, body, error, transport }
          ‚Ēā
          ‚Ėľ
   one correction hook  (PostToolUse/PostToolUseFailure) inspects the envelope
     + protocol-aware checks layered in

  Why per-protocol thin tools, not one mega-tool. The call shapes genuinely
  differ: REST is method+path+query+body; GraphQL is a query string + variables
  over a single POST endpoint; gRPC is service/method + a message over HTTP/2
  with a proto contract. Cramming those into one union schema makes the model's
  job harder, and more importantly the failure modes differ in ways the
  correction hook must know about:
  - GraphQL returns HTTP 200 with an errors[] array even when the operation
  failed. A naive "non-2xx = error" check misses every GraphQL failure. This is
  exactly the "looks successful but isn't" case where a protocol-aware hook
  earns its keep.
  - gRPC has its own status codes (and trailers) distinct from HTTP status.

  So each adapter normalizes into one envelope shape (so the hook has a single
  thing to read), but contributes its own structural checks. That keeps the
  self-correction trigger correct across transports instead of REST-shaped.

  Transport is learned, never assumed. Ingestion detects it from the shape of
  what's pasted: REST docs (endpoints + verbs), a GraphQL SDL (type Query { ... 
  }, single endpoint), or a .proto / service X { rpc ... } for gRPC. It records
  transport: rest|graphql|grpc plus connection details in integration.md, and
  the use loop selects the matching adapter from that config. Nothing

     Context

     This is a 3-hour engineering take-home ("Autointegrate", brief in

     Architecture recap

‚Źļ Sure. That paragraph is four design decisions packed together. Let me unpack
  each.

  The setup: what "probing" is

  When the operator pastes a new API's docs, we don't just trust the docs. We
  spin up a burst of short-lived helper agents that actually call the live API
  to check what's real (does the list endpoint cap results? what status values
  actually appear? do the documented fields really show up?). Those helpers are
  the "probers." This only happens once, on first contact with a new API, and
  the result is written into ./memory so later sessions don't re-probe.

  Decision 1: nested query() instead of SDK options.agents

  The Agent SDK gives you two ways to run sub-agents:
  - options.agents (model-driven): you declare named sub-agents, and the main 
  model decides when to spawn them by calling the built-in Agent tool. The
  choreography is up to the LLM.
  - Nested query() (code-driven): our own handler code calls query() again,
  directly, N times. We control exactly which probers run, with what prompt, on
  what model.

  We picked the second. Two reasons:
  - Determinism. Ingestion should behave the same on every run (the brief
  literally says "your reasoning should not" change run to run). If the main
  model decides whether/how to fan out, that varies. If our code runs a fixed
  set of probers, it doesn't.
  - Cost attribution. Each query() call returns its own ResultMessage with
  total_cost_usd. By running probers as separate query() calls, we can sum their
  cost and report "ingestion cost ~$X" separately from "answering questions
  cost ~$Y" in the decision record. If probers were spawned by the main model
  via the Agent tool, their cost folds into the parent's total and is hard to
  separate.

  So a prober here is just "another query() we launch from inside the
  ingest_spec tool's handler," not an SDK-managed sub-agent.

  Decision 2: probers are read-only; they return findings, they don't write 
  memory

  Each prober is given the transport tools (to call the API) plus Read (to see
  the draft config), but not Edit/Write. Its job is to investigate and then end
  its turn with a structured block of findings, for example:

  { "endpoint": "<list>", "default_limit_observed": 25, "total_via_paging": 240,
    "status_values_seen": ["new","orientation","hired","rejected","withdrawn"],
    "fields_missing_vs_spec": [], "extra_fields": ["medical_card_expires_at"] }

  The SDK returns a sub-query()'s final message verbatim to the caller, so the
  ingest_spec handler reads that JSON back out and collects all the probers'
  findings.

  Decision 3: the ingest_spec handler is the single memory writer

  After the probers report back, only the ingest_spec handler writes to ./memory
  (it drafts integration.md, merges the findings, creates each profile.md,
  seeds corrections.md, updates INDEX.md). Why centralize writing in one place:
  - No write races. If five probers all tried to Edit the same integration.md in
  parallel, they'd clobber each other (read-modify-write on one file from five
  processes at once). Making them read-only and giving one writer the merge step
  removes that entire class of bug.
  - One audit chokepoint. Every memory write then flows through the single
  PreToolUse memory guard, so the integrity rules (no clobbering
  INDEX.md/integration.md, no writes escaping memory/) are enforced in exactly
  one path instead of five.

  This is the classic "many readers, one writer" pattern: parallelize the slow
  investigation, serialize the delicate write.

  Decision 4: Promise.all vs the sequential fallback

  The probers are independent, so by default we launch them concurrently with
  Promise.all([...]) and wait for all to finish (fast: wall-clock = the slowest
  prober, not the sum). The "sequential fallback" is the degraded mode: if
  parallel probing is flaky or we're tight on the budget cap, we run them one at
  a time in a for loop instead. Same probers, same findings, same merged
  output, just slower and easier to reason about. It's the safety valve, not the
  default.

  The whole flow in one line

  Operator pastes docs ‚Üí ingest_spec handler launches several read-only prober
  query() calls (in parallel) that hit the live API and each return findings
  JSON ‚Üí the handler merges all findings and is the only thing that writes the
  result into ./memory, through the one guard.

  The net effect: fast, repeatable, cost-measurable probing with no file races
  and a single place where memory can be written.

  the path from the envelope's meta: apiSlug ‚Üí memory/apis/<slug>/, and if

  scope) else integration.md, and always names corrections.md for the log entry.
  That [SELF-CORRECTION SIGNAL] text is appended to the tool result the model
  reads; the model then edits those exact files (through the memoryGuard).
